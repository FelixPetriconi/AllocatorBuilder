1 Allocator Builder {#mainpage}
2 =================
3 
4 A highly composable, policy based C++ allocator.
5 
6 The layout idea of the library is was presented by [Andrei Alexandrescu](http://erdani.com/) at the [C++ and Beyond 2013](http://cppandbeyond.com/) seminar and at the [CppCon 2015](http://cppcon2015.sched.org/event/95b6c3282248b7e2595c5c3182d7652b).
7 
8 The background behind the idea is to compensate the main problem of malloc and the other current standard allocators, separation of memory pointer and the allocated size. This makes it very difficult for all kind of memory management to handle in a fast way memory allocations and especially deallocations, because the memory manager must find the corresponding length of the pointer that shall be free.
9 As well all memory handler are currently general purpose handler. This policy based approach allows to create handlers for special purpose.
10 Additionally all users of manually raw allocated memory have to store the size anyway to ensure that no access beyond the length of the allocated buffer takes place.
11 
12 An other idea behind this allocator library is, that one can compose for every use case a special designed one.
13 Example use cases:
14  * Collect statistic information about the memory usage profile.
15  * Apply guards to memory allocated blocks to detect buffer under- or overflows, even in release mode of the compiled application.
16  * Wait free allocations in a single threaded environment
17 
18 So the approach is, every allocator returns such a block:
19 ~~~
20 struct block {
21  void *ptr;
22  size_t length;
23 };
24 ~~~
25 
26 And a request goes this way:
27 ~~~
28 auto myMemBlock = allocator.allocate(42);
29 ~~~
30 
31 Motivation
32 ----------
33 Raw memory is temporarily needed inside a function, 64-128 bytes per allocation. The fastest way would be to use ::alloca(); getting the memory from the stack. But this memory cannot be freed explicitly. But with the combination of free-list these memory block could be recycled. This approach would be must faster than getting the memory from the heap. The order of allocations and de-allocations can happen in any order.
34 
35 So the code could look like this
36 ~~~
37 using StackRecycler = alb::free_list<alb::stack_allocator<16384>, 64, 128, 128>;
38 
39 StackRecycler localAllocator;
40 auto m1 = localAllocator.allocate(64);
41 auto m2 = localAllocator.allocate(128);
42 
43 localAllocator.deallocate(m1); // This freed memory can now we reused with calling the next allocation
44 localAllocator.deallocate(m2);
45 ~~~
46 
47 A more advanced allocator with different sized buckets as one are used in [jemalloc](http://www.canonware.com/jemalloc/) would look like:
48 ~~~
49 // This defines a FreeList that is later configured by the bucketizer to its size
50 using FList = freelist<mallocator, DynamicSetSize, DynamicSetSize>;
51 
52 // All allocation requests up to 3584 bytes are handled by the cascade of FLists.
53 // Sizes from 3584 till 4MB are handled by the Heap and all beyond that are forwarded
54 // directly to the normal OS
55 using AdvancedAllocator = segregator<
56  8, freelist<mallocator, 0, 8>, segregator<
57  128, bucketizer<FList, 1, 128, 16>, segregator<
58  256, bucketizer<FList, 129, 256, 32>, Segegator<
59  512, bucketizer<FList, 257, 512, 64>, Segegator<
60  1024, bucketizer<FList, 513, 1024, 128>, Segegator<
61  2048, bucketizer<FList, 1025, 2048, 256>, Segegator<
62  3584, bucketizer<FList, 2049, 3584, 512>, Segegator<
63  4072 * 1024, cascading_allocator<Heap<mallocator, 1018, 4096>>, mallocator
64  >
65  >
66  >
67  >
68  >
69  >
70  >>;
71 ~~~

74 Allocator Overview
75 ------------------
76 
77 |Allocator |Description |
78 ---------------------------|----------------------------------------------------------------------------
79 | affix_allocator | Allows to automatically pre- and sufix allocated regions. |
80 | allocator_with_stats | An allocator that collects a configured number of statistic information, like number of allocated bytes, number of successful expansions and high tide |
81 | bucketizer | Manages a bunch of Allocators with increasing bucket size |
82 | fallback_allocator | Either the default Allocator can handle a request, otherwise it is passed to a fall-back Allocator |
83 | (aligned_)mallocator | Provides and interface to systems ::malloc(), the aligned variant allocates according to a given alignment |
84 | null_allocator | An Null allocator |
85 | segregator | Separates allocation requests depending on a threshold to Allocator A or B |
86 | (shared_)freelist | Manages a list of freed memory blocks in a list for faster re-usage. (The Shared variant is thread safe) |
87 | (shared_)cascading_allocator | Manages in a thread safe way Allocators and automatically creates a new one when the previous are out of memory. (The Shared variant is thread safe, but it needs further improvements, because it does not frees unused allocators) |
88 | (shared_)heap | A heap block based heap. (The Shared variant is thread safe manner with minimal overhead and as far as possible in a lock-free way.) |
89 | stack_allocator | Provides a memory access, taken from the stack |

134 Installation Win
135 ----------------
136  * Have boost installed and the standard libs be build, install into D:\boost_1_55_0 and set in the CMakeLists.txt the correct path
137  * Clone into e.g. D:\misc\AllocatorBuilder
138  * Create a build folder, eg D:\misc\alb_build
139  * Open a command prompt in that alb_build folder
140  * Have CMake in the path
141  * Execute cmake -G "Visual Studio 14 2015 Win64" ..\alb_build
142  * Open created solution in .\alb_build\AllocatorBuilder.sln
143  * Compile and run all test (if necessary add D:\boost_1_55_0\stage\lib to search path)
