1 Allocator Builder {#mainpage}
2 =================
3 
4 A highly composable, policy based C++ allocator based on ideas from [Andrei Alexandrescu](http://erdani.com/), presented at the [C++ and Beyond 2013](http://cppandbeyond.com/) seminar.
5 
6 The background behind the idea is to compensate the main problem of malloc and the other standard allocators, separation of memory pointer and the allocated size. This makes it very difficult for all kind of memory management to handle in a fast way memory allocations and deallocations.
7 All users of manually allocated memory have to store the size anyway to ensure that no access beyond the length of the allocated buffer takes place.
8 
9 A second idea behind this allocator library is, that one can compose for every use case a special designed one.
10 Example use cases:
11  * Collect statistic information about the memory usage profile.
12  * Apply guards to memory allocated blocks to detect buffer under- or overflows, even in release mode of the compiled application.
13  * Wait free allocations in a single threaded environment
14 
15 So the approach is, every allocator returns such a Block
16 ~~~
17 struct block {
18  void* ptr;
19  size_t length;
20 };
21 ~~~
22 
23 And request goes this way:
24 ~~~
25 auto myMemBlock = allocator.allocate(42);
26 ~~~
27 
28 Motivation
29 ----------
30 Raw memory is temporary needed inside a method. Most of the time the amount memory would fit on the stack and so :alloca() is ones friend. But in seldom cases more is needed and so :malloc() must be used. (Allocation on the stack is much faster because it a wait-free operation and in many cases the allocated memory is much more cache friendly.)
31 
32 So the code could look like this
33 ~~~
34 const int STACK_THRESHOLD = 1024;
35 int neededBytes = 42
36 bool wouldFitOnTheStack = neededBytes < STACK_THRESHOLD;
37 char* p = wouldFitOnStack? : (char*):alloca(neededBytes) : nullptr;
38 
39 std::unique_ptr<char[]> heapAllocated(!wouldFitOnStack? new char[neededBytes] : nullptr);
40 
41 if (!p) {
42  p = heapAllocated.get();
43 }
44 
45 // ... work with p[0..neededBytes-1]
46 ~~~
47 
48 Everybody would agree that this is not nice! So what if one could encapsulate this into something like:
49 ~~~
50 const int STACK_THRESHOLD = 1024;
51 int neededBytes = 42
52 
53 typedef fallback_allocator<stack_allocator<STACK_THRESHOLD>, mallocator> LocalAllocator;
54 LocalAllocator localAllocator;
55 
56 auto block = localAllocator.allocate(neededBytes);
57 auto p = static_cast<char*>(block.ptr);
58 
59 // ... work with p[0..neededBytes-1]
60 
61 localAllocator.deallocate(block); // better to be deleted by a scope finalizer
62 ~~~
63 So, isn't this much cleaner?
64 
65 
66 A more advanced allocator as it is used in [jmalloc](http://www.canonware.com/jemalloc/) would look like:
67 ~~~
68 // This defines a FreeList that is later configured by the bucketizer to its size
69 typedef freelist<mallocator, DynamicSetSize, DynamicSetSize> FList;
70 
71 // All allocation requests up to 3584 bytes are handled by the cascade of FLists.
72 // Sizes from 3584 till 4MB are handled by the Heap and all beyond that are forwarded
73 // directly to the normal OS
74 typedef segregator<
75  8, freelist<mallocator, 0, 8>, segregator<
76  128, bucketizer<FList, 1, 128, 16>, segregator<
77  256, bucketizer<FList, 129, 256, 32>, Segegator<
78  512, bucketizer<FList, 257, 512, 64>, Segegator<
79  1024, bucketizer<FList, 513, 1024, 128>, Segegator<
80  2048, bucketizer<FList, 1025, 2048, 256>, Segegator<
81  3584, bucketizer<FList, 2049, 3584, 512>, Segegator<
82  4072 * 1024, cascading_allocator<Heap<mallocator, 1018, 4096>>, mallocator
83  >
84  >
85  >
86  >
87  >
88  >
89  >>AdvancedAllocator;
90 ~~~

93 Allocator Overview
94 ------------------
95 
96 |Allocator |Description |
97 ---------------------------|----------------------------------------------------------------------------
98 | affix_allocator | Allows to automatically pre- and sufix allocated regions. |
99 | allocator_with_stats | An allocator that collects a configured number of statistic information, like number of allocated bytes, number of successful expansions and high tide |
100 | bucketizer | Manages a bunch of Allocators with increasing bucket size |
101 | fallback_allocator | Either the default Allocator can handle a request, otherwise it is passed to a fall-back Allocator |
102 | (aligned_)mallocator | Provides and interface to systems ::malloc(), the aligned variant allocates according to a given alignment |
103 | segregator | Separates allocation requests depending on a threshold to Allocator A or B |
104 | (ahared_)freelist | Manages a list of freed memory blocks in a list for faster re-usage. (The Shared variant is thread safe manner) |
105 | (shared_)cascading_allocator | Manages in a thread safe way Allocators and automatically creates a new one when the previous are out of memory. (The Shared variant is thread safe manner) |
106 | (shared_)heap | A heap block based heap. (The Shared variant is thread safe manner with minimal overhead and as far as possible in a lock-free way.) |
107 | stack_allocator | Provides a memory access, taken from the stack |

149 Installation Win
150 ----------------
151  * Have boost installed and the standard libs be build, install into D:\boost_1_55_0
152  * Clone into e.g. D:\misc\AllocatorBuilder
153  * Create a build folder, eg D:\misc\alb_build
154  * Open a command prompt in that alb_build folder
155  * Have CMake in the path
156  * Execute cmake -G "Visual Studio 11 Win64" ..\alb_build
157  * Open created solution in .\alb_build\AllocatorBuilder.sln
